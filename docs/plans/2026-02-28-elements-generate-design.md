# Custom Elements Generate — Design Document

**Date:** 2026-02-28
**Status:** Approved

---

## Goal

Add a `make elements-generate` command that auto-derives render-ready versions of every custom element from the existing canvas-v2 source. The generated output is consumed by `@selorax/renderer` so custom elements work in the storefront (port 3003) and preview server (port 3004) without any manual duplication.

## Problem

Custom elements (`countdown`, `marquee`, `accordion`, etc.) are stored in STDB as `node_type: "component"` with `settings.customType = "countdown"` and `settings.data = { ... }`. The renderer's `RenderComponent` branch only handles CDN-loaded ESM components (`component_url`). When `component_url` is `null` (which it is for all custom elements), it renders a gray "Component URL missing" placeholder.

The source components live in `apps/canvas-v2/src/components/custom-registry/` and are tightly coupled to the editor (they accept `{ element: FunnelElement, onUpdate?, isPreview? }` — canvas-v2-specific types). They cannot be imported directly from the renderer package (wrong dependency direction, bundled with editor deps).

## Architecture

```
apps/canvas-v2/src/components/custom-registry/   ← source of truth (editor full components)
         │
         ▼  make elements-generate
scripts/elements-generate.ts                      ← generator script (run via tsx)
         │
         ▼ bundles each element with esbuild
packages/renderer/src/elements/                   ← generated output (committed to repo)
  ├── countdown.js
  ├── marquee.js
  ├── accordion.js
  ├── ...
  └── registry.ts                                 ← generated: customType → () => import('./X.js')
         │
         ▼ imported by
packages/renderer/src/PageRenderer.tsx            ← RenderCustomElement branch added
```

## Approach: esbuild Adapter Bundle

For each element source file, the generator:

1. **Reads the `CustomComponentDef` export** to get `component` reference and `defaultData`
2. **Writes a tiny adapter entry file** (temp, in memory / tmpdir) that:
   - Imports the original component
   - Exports a `default` function with the renderer's `{ data, style }` prop contract
   - Calls the original with `isPreview={true}` and adapts props
3. **Runs esbuild** with `external: ['react', 'react-dom']` to produce a standalone `.js` bundle
4. **Writes output** to `packages/renderer/src/elements/<customType>.js`
5. **Generates `registry.ts`** mapping all `customType` strings → lazy `() => import('./X.js')` calls

esbuild's tree-shaking naturally drops `onUpdate`, `CustomComponentDef` export, and any code inside `if (!isPreview)` guards when `isPreview` is constant `true`.

## Prop Contract

**Editor components accept:**
```typescript
{ element: FunnelElement, onUpdate?: (...) => void, isPreview?: boolean, deviceView?: string }
// Read data via: element.data.X, element.style, element.className
```

**Renderer components accept:**
```typescript
{ data: Record<string, any>, style?: React.CSSProperties, className?: string }
// data === element.data, style === element.style
```

**Adapter template** (generated per element):
```typescript
import { CountdownComponent } from '../../apps/canvas-v2/src/components/custom-registry/Countdown';

const fakeElement = (data: any, style?: any) => ({
  id: '__render__', type: 'countdown', data, style: style ?? {}, className: '',
  children: [], settings: {}, props: {},
});

export default function CountdownRender({ data, style }: { data: any; style?: any }) {
  return <CountdownComponent element={fakeElement(data, style)} isPreview={true} />;
}
```

## Registry Shape

Generated `packages/renderer/src/elements/registry.ts`:
```typescript
// AUTO-GENERATED by scripts/elements-generate.ts — do not edit manually
// Run: make elements-generate

export const CUSTOM_ELEMENT_REGISTRY: Record<string, () => Promise<{ default: React.ComponentType<any> }>> = {
  countdown:         () => import('./countdown.js'),
  marquee:           () => import('./marquee.js'),
  accordion:         () => import('./accordion.js'),
  list:              () => import('./list.js'),
  detail_list:       () => import('./list_details.js'),
  carousel:          () => import('./image_carousel.js'),
  imageCarousel:     () => import('./image_carousel.js'),   // alias
  gallery:           () => import('./gallery.js'),
  // ... all aliases from CUSTOM_BLOCKS
};
```

## Renderer Change

`RenderComponent` in `PageRenderer.tsx` gains a `customType` branch:

```typescript
function RenderComponent({ node, styles, data }: any) {
  const customType = node.settings?.customType;

  // Branch A: custom element from generated registry
  if (customType) {
    return <RenderCustomElement customType={customType} data={node.settings?.data ?? {}} styles={styles} />;
  }

  // Branch B: CDN ESM component (existing behaviour)
  if (!node.url) return <div style={{ ...styles, minHeight: 60 }}>Component URL missing</div>;
  // ... existing dynamic import logic
}

function RenderCustomElement({ customType, data, styles }: any) {
  const [Comp, setComp] = useState<any>(null);
  useEffect(() => {
    CUSTOM_ELEMENT_REGISTRY[customType]?.()
      .then(m => setComp(() => m.default))
      .catch(() => setComp(null));
  }, [customType]);

  if (!Comp) return <div style={{ ...styles, minHeight: 60 }} />;
  return <div style={styles}><Comp data={data} /></div>;
}
```

## Generator Script

**Location:** `scripts/elements-generate.ts` (at monorepo root)

**Run via:** `tsx scripts/elements-generate.ts`

**Steps:**
1. Import `CUSTOM_BLOCKS` from `apps/canvas-v2/src/components/custom-registry.tsx`
2. Deduplicate: multiple keys → same component file (e.g., `carousel`, `imageCarousel`, `product_carousel` all → `ImageCarousel`)
3. For each unique component file, write adapter entry to `os.tmpdir()`
4. Run `esbuild` with:
   - `entryPoints`: adapter file
   - `bundle: true`, `format: 'esm'`, `jsx: 'automatic'`
   - `external: ['react', 'react-dom', 'react/jsx-runtime']`
   - `outfile`: `packages/renderer/src/elements/<slug>.js`
5. Write `packages/renderer/src/elements/registry.ts` with all alias mappings
6. Print summary: `✓ Generated N element bundles`

## Makefile Target

```makefile
elements-generate:
    tsx scripts/elements-generate.ts
    @echo "✓ Custom element bundles written to packages/renderer/src/elements/"
```

## What Gets Committed

- `scripts/elements-generate.ts` — generator (source, always committed)
- `packages/renderer/src/elements/*.js` — generated bundles (**committed** — same as STDB generated bindings)
- `packages/renderer/src/elements/registry.ts` — generated registry (**committed**)

Generated files are committed (not gitignored) so storefront/preview builds don't require running the generator. Add a CI check that re-runs the generator and verifies no diff, same pattern as other generated code.

## Key Constraints

1. **No cross-package runtime imports** — adapter uses relative path to canvas-v2 source only during the build step (esbuild resolves it); the output `.js` is self-contained
2. **`react` externalized** — bundles don't bundle React; renderer's host React instance is used
3. **Aliases handled in registry** — `carousel`, `imageCarousel`, `product_carousel` all map to the same `.js`; registry has all alias entries
4. **`DynamicIcon` from `utils.tsx`** is needed at render time — it's included in the bundle (not editor-only)
5. **EditableText is excluded** — when `isPreview={true}` is passed, `EditableText`'s editable branch is dead code; esbuild tree-shakes it away (or it renders as plain `<span>`)
6. **`esbuild` added as dev dep** to `scripts/package.json` or monorepo root `devDependencies`
